2 Caveman Mode automatique

Compresse les réponses de Claude (~65% tokens en moins). Activé par défaut à chaque session.

Commandes

CommandeEffet
/cavemanMode full (défaut) — fragments OK, articles supprimés
/caveman liteMoins compressé — phrases complètes, juste le filler supprimé
/caveman ultraCompression maximale — abréviations (DB, auth, fn, impl)
stop cavemanRevenir au mode normal verbose
/caveman-helpCarte de référence rapide
/caveman-commitMessage de commit ultra-court
/caveman-reviewReview de code en 1 ligne par commentaire

Exemple

ModeRéponse à "Pourquoi le composant re-render ?"
NormalLe problème vient du fait que vous créez une nouvelle référence d'objet à chaque rendu. Je vous recommande d'utiliser useMemo pour mémoriser la valeur.
FullNew object ref each render. Inline object prop = new ref = re-render. Wrap in useMemo.
UltraInline obj prop → new ref → re-render. useMemo.

Statusline

Badge visible dans la barre de statut : [CAVEMAN] ou [CAVEMAN:ULTRA]. Configuré dans ~/.claude/settings.json.

3 Deploy.sh commande manuelle

Déployer un module en 1 commande au lieu de copier fichier par fichier via SSH.

Syntaxe

cd /c/odoo18/custom-addons
./scripts/deploy.sh <module> <45|43>

Déployer sur TEST (serveur 45)

./scripts/deploy.sh pos_dashboard 45

Déployer sur PROD (serveur 43)

./scripts/deploy.sh pos_dashboard 43

JAMAIS de restart sur 43 pendant les heures de travail ! 115 utilisateurs + 40 sessions PdV actives. Le script copie les fichiers mais ne redémarre PAS. Le restart est à faire le soir.

# Sur le serveur (via SSH)
docker exec odoo-dev /opt/odoo/odoo-bin \
  -c /etc/odoo/odoo.conf \

  --stop-after-init

Ce que le script fait

  1. Vérifie que le module existe localement avec un __manifest__.py
  2. Demande confirmation si c'est la PROD (43)
  3. Supprime l'ancien module sur le serveur
  4. Copie via tar+SSH (exclut __pycache__, .git, node_modules)
  5. Restart automatique si TEST (45), message "ce soir" si PROD (43)
  6. Met à jour deployed_versions.json et deploy.log

7 Tests E2E — Playwright & Puppeteer nouveau

Tests navigateur automatisés pour valider l'UI, composants OWL/JS, POS — tout ce que XML-RPC ne peut pas couvrir.

RESTRICTION ABSOLUE : Tests E2E = serveur 45 (test) UNIQUEMENT. Jamais sur 43 (prod) ni odoo.sopromer.mg. Les config.py / config.js bloquent automatiquement toute tentative vers la prod.

Playwright vs Puppeteer

Playwright Python PRÉFÉRÉPuppeteer Node.js LEGACY
LangagePython (même qu'Odoo)JavaScript
Attente élémentsAuto-wait intégréManuel (delay(2000)) — flaky
SélecteursPar texte : page.get_by_role("button", name="Confirmer")CSS : page.$('.o_btn')
Helpers Odoofill_many2one, click_status_bar, rpc_callBasiques (login, navigate, screenshot)
DebugTrace viewer (vidéo + réseau + DOM)Screenshots uniquement
StabilitéTrès stableTests flaky fréquents (timing)
Règle : Tout nouveau test = Playwright Python. Puppeteer reste pour les tests existants.

8 Template CI/CD à copier

Fichier GitHub Actions prêt à copier dans chaque module pour du lint automatique au push.

Comment ça marche

Tu fais

GitHub lance

→
✓ OK → badge vert
✗ Erreur → annotations

Ce que le workflow vérifie

JobCheckBloquant ?
Python QualityRuff lint — erreurs de code PythonOUI
Ruff format — style de code Python (espaces, indentation)OUI
Odoo StructureManifeste — version format 18.0.X.Y.ZOUI
Manifeste — license, author, depends présentswarning
XML valide — pas d'erreur de syntaxe dans les vuesOUI
Sécurité — ir.model.access.csv si nouveaux modèleswarning
Odoo Structure (suite)Imports __init__.py — tous les .py importésOUI
CSV header — format correctwarning
Repo HygienePas de __pycache__/ ou .pyc commitésOUI
Pas de fichiers secrets (.env, .key, .pem)OUI
.gitignore présentwarning
README.md présentwarning
Bloquant vs Warning : Un check bloquant fait échouer le CI (croix rouge ✗). Un warning affiche un avertissement mais ne bloque pas.

Comment activer pour un module

# Créer le dossier workflows dans le module
mkdir -p pos_dashboard/.github/workflows

# Copier le template
cp scripts/ci-lint.yml pos_dashboard/.github/workflows/ci.yml

# Commit + push
cd pos_dashboard
git add .github/
git commit -m "ci: add CI workflow — lint, structure, hygiene"
git push
Après ça, chaque push sur GitHub lance automatiquement les 3 jobs de vérification. Résultats visibles dans l'onglet Actions du repo.

Voir les résultats

# Sur GitHub : repo → onglet Actions
✓ CI Odoo Module — main — 25s ago       ← tout OK
✗ CI Odoo Module — main — 5min ago      ← cliquer pour voir les erreurs

# En CLI avec gh
gh run list                              # voir les derniers runs
gh run view --log-failed                # voir les erreurs du dernier run

11 Workflow quotidien

Journée type

  1. Ouvrir Claude Code dans C:\odoo18\custom-addons\
    → CLAUDE.md chargé, caveman actif, mémoires chargées — automatique
  2. Demander une tâche
    "Ajoute un champ date_livraison sur sale.order"
    → Claude utilise odoo-backend, ruff lint auto sur chaque fichier modifié
  3. Déployer sur TEST
    ./scripts/deploy.sh mon_module 45
    → Copie + restart auto + log + versions tracker mis à jour
  4. Tester sur le serveur 45
    → Vérifier manuellement ou demander à odoo-tester
  5. Déployer sur PROD (quand validé)
    ./scripts/deploy.sh mon_module 43
    → Copie seulement, pas de restart
  6. Le soir — restart PROD
    ssh odoo@192.73.0.43
    docker restart odoo-dev
    → Module actif le lendemain matin
  7. Vérifier en prod le lendemain
    → Tester que tout fonctionne sur odoo.sopromer.mg
